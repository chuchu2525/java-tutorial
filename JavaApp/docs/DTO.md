# DTO（Data Transfer Object）とは

レイヤーやプロセスの間で **データをまとめて渡すための入れ物** を表すクラスです。Spring Boot の図書デモでは `com.example.demo.library.dto` パッケージに、フォーム用・API用の DTO が置かれています（全体構成は [ARCHITECTURE.md](ARCHITECTURE.md) も参照）。

---

## 1. 一言でいうと

**DTO = アプリの「区切り」（画面・API・別サービス）をまたぐときに、運ぶデータの形をはっきり決めた、シンプルなデータ入れ物。**

---

## 2. たとえ話：伝票・届け出用紙

図書館で「本を借りたい」と言うとき、利用者が書くのは名前・会員番号・本のID など **決まった欄だけ** です。

- **DTO** … その「欄が決まった用紙」。ここに書いた内容だけが、次の担当（プログラムの別の部分）へ渡る。
- つまり **「このやりとりでは、こういう項目だけを運ぶ」** という契約を型で表すクラスです。

---

## 3. Webアプリでの典型的な流れ

1. **ブラウザ**が JSON やフォームでデータを送る。
2. **コントローラ**が、それを **DTO** に詰め替える（Spring ならバインドで自動になることが多い）。
3. **サービス／リポジトリ**が、必要なら **DB用のオブジェクト（エンティティ）** に変換して保存する。
4. 返すときも、必要なら **別の DTO** にしてブラウザへ返す。

「外から入ってきたデータの形」と「DBの中の形」は **同じである必要はない** ので、その間をつなぐのが DTO です。

---

## 4. エンティティとの違い

| | **エンティティ** | **DTO** |
|---|------------------|---------|
| イメージ | DBの1行に近い「社内の本当のデータ」 | 画面・API用に切り出した「届け出・返信用」 |
| 中身 | ID、作成日時、他テーブル参照など DB都合のものが乗りがち | 今回の画面/APIに必要なフィールドだけ |
| 主な目的 | **永続化（保存）** | **運ぶ・見せる・受け取る** |

### DTO を挟む主な理由

- **セキュリティ** … パスワードや内部だけのIDなど、クライアントに出したくない項目を、返すDTOからそもそも除外できる。
- **APIの安定** … DBのカラムを増やしても、返すDTOを変えなければフロントが壊れにくい。
- **バリデーション** … 「このフォームではタイトルは必須」などを、DTOのフィールドにルール（Bean Validation 等）を付けて表現しやすい。

---

## 5. Request DTO と Response DTO

よく次のように分けます。

- **Request DTO（入力用）** … ユーザーが送ってきた内容を受け取る形（例: 本を追加するフォーム、貸出リクエスト）。
- **Response DTO（出力用）** … 一覧に出す情報など、返したい形に絞ったもの。

同じ「本」でも、追加用と一覧用で **フィールドの数や名前が違ってよい** のが普通です。

---

## 6. コード上のイメージ

だいたい **フィールド＋getter/setter（または Java `record`）** で、**ビジネスロジックはほぼ書かない** クラスです。

- 「計算する」「DBに保存する」は **サービス** の仕事。
- DTO は **データの入れ物**。

学習用コードを読むときは、**画面のURL → コントローラのメソッド → 引数の型** を追うと、その引数が DTO かどうかが分かりやすいです。
